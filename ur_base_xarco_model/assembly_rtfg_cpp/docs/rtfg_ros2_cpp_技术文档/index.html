1. 研究背景与目标

assembly_rtfg_cpp 是 MATLAB realtime_trajectory_fit_gui 的 ROS2 C++ 第一版重构包。它不替换现有 assembly_rviz_tcp_control 仿真包，而是在当前工作区内新增一个独立 ament C++ 包，用 MATLAB 版 environmental_model/assembly_with_block_with_basin.urdf 作为模型基线。

重构的核心目标包括：

  • 脱 MATLAB 依赖：将原本绑定 MATLAB Runtime 的轨迹拟合链路完整迁移到原生 ROS2 C++ 环境
  • 服务化接口：以 ROS2 service/topic 替代 MATLAB GUI 按钮回调，实现非实时模块化调用
  • 可视化接入：通过 RViz PoseArray、MarkerArray 和 MoveIt DisplayTrajectory 替代 MATLAB 绘图
  • 性能对标：在保留 MATLAB/MEX 算法精度的前提下，验证 ROS2 C++ 首版的性能基线

该包提供完整的轨迹生成、TCP 目标姿态构造、IK 拟合、碰撞审计、回放轨迹生成、RViz/MoveIt 展示和 fake ros2_control 执行入口。

3. 对比效果 — 性能初步评估

3.1 当前性能基线

第一版已经完成脱 MATLAB、ROS2 服务化和 RViz/MoveIt 可视化接入。默认 YAML 参数可以生成完整关节轨迹、TCP path 和碰撞审计结果。

关键性能结论（必须按实测数字）： 当前 ROS2 C++ 第一版还没有超过历史 MEX 基线。主要原因不是 C++ 本身慢，而是第一版为了稳定复刻 MATLAB 行为，保留了更保守的多阶段 DLS、多种子搜索、候选级碰撞门控，以及 ROS2 service/topic 的序列化和可视化数据发布。

4. 结果分析

4.1 性能瓶颈定位

  • 多阶段 DLS 求解：每姿态最多尝试 ~120 个种子 × 4 组权重 = ~480 次 IK 调用，累积开销大
  • 候选级碰撞门控：每个候选解均经过 FCL 距离查询，O(N^2) 自碰撞对检查
  • ROS2 中间件开销：service 请求/响应的序列化、topic 可视化的数据发布增加了延迟
  • 线程模型：当前为单线程同步编排，未利用多核并行加速

4.2 当前限制

  • 默认参数集下 tool-basin 碰撞穿透未解决
  • 性能尚未达到 MEX 基线水平
  • 缺乏实时执行保障（无 RT 优先级、无 deadline 调度）
  • 未接入真实 ros2_control 硬件接口

5. 结论与改进效果

5.1 当前阶段总结

第一版已完成以下里程碑：

  • 完整的 C++ 算法链路：轨迹生成 → IK 求解 → 碰撞审计 → 回放轨迹生成
  • ROS2 服务化封装：三个核心 service 接口
  • RViz/MoveIt 可视化集成
  • fake ros2_control 执行通道
  • benchmark 测试框架
  • 安全参数集已验证可通过碰撞检测

5.3 快速运行

启动完整仿真（含 RViz）：

source /opt/ros/humble/setup.bash
source /home/liuxiaopeng/ws_moveit2/install/setup.bash
source /home/liuxiaopeng/ur10_conrtol/ur_base_xarco_model/install/setup.bash
ros2 launch assembly_rtfg_cpp rtfg_sim.launch.py launch_rviz:=true

无 RViz smoke 测试：

ros2 launch assembly_rtfg_cpp rtfg_sim.launch.py launch_rviz:=false launch_moveit:=false

读取配置：

ros2 service call /rtfg/load_config assembly_rtfg_cpp/srv/LoadConfig "{yaml_path: ''}"

运行 benchmark：

python3 install/assembly_rtfg_cpp/share/assembly_rtfg_cpp/benchmarks/rtfg_benchmark.py
