10. ROS2 接口字段与排障手册

Service 字段说明、Topic 检查、Launch/Controller/MoveIt 常见问题、最小排障顺序

1. 研究背景与目标

assembly_rtfg_cpp 包通过 ROS2 Service 接口对外暴露轨迹拟合能力。标准化的接口规范是保证调用端（GUI / 仿真桥 / 实车节点）与 Solver 节点之间正确协作的前提。同时，Solver 的启动依赖 URDF 加载、Controller 激活、MoveIt 配置等多个环节，任一环节出问题都可能导致 Service 不可用或结果异常。

本章的核心目标：

  • 精确定义 LoadConfig 和 FitPreview 两个 Service 的请求/响应字段语义；
  • 给出 Topic / Controller / MoveIt 的在线检查命令与期望输出；
  • 收录已知常见问题的根因与修复方法；
  • 提供最小排障顺序，避免在基础环境问题上浪费 IK 算法调试时间。

2.4 设计原则

initial_q 单一数据源：initial_q 来自 URDF ros2_control block，而不是 YAML。这样保证模型初始姿态只在 URDF 中维护一份，避免 YAML-URDF 两份数据不一致导致的关节初始偏移。
Namespace 隔离：Controller Manager 已放入 /rtfg namespace，避免与工作区已有仿真或真实机器人 controller 冲突。
失败诊断路径：即使求解失败，也返回 trajectory 用于 RViz 可视化和诊断。只有 success=true 的结果才适合 /rtfg/execute_cached 执行。

4. 结果分析：接口设计与排障策略的合理性

4.1 字段设计的语义正交性

LoadConfig 和 FitPreview 的分离设计保证了关注点分离：

  • LoadConfig 负责环境配置加载，返回的字段是静态几何参数和初始关节角，不涉及求解状态。
  • FitPreview 负责轨迹拟合，同时接收作业参数（theta_deg、depth 等）和当前机器人状态（current_q）。
  • ExecuteCached 负责执行，仅接受 success=true 的结果。

initial_q 来自 URDF 而非 YAML 的设计尤为关键。YAML 中的初始姿态可能在手动编辑时偏离 URDF，两份数据的不一致会导致启动时关节角跳变。将单一数据源锁定在 URDF 的 ros2_control block，消除了这个风险。

4.2 Timing 字段的瓶颈定位价值

FitPreview 响应中包含 4 个 timing 字段，构成一个多层性能分析工具：

  • timing_ik_total_s — IK 求解器的原始计算开销。高则说明候选种子搜索空间大或 DLS 迭代收敛慢。
  • timing_collision_total_s — FCL 碰撞检测开销。高则说明自碰撞对数量多或料斗 Box 距离查询密集。
  • timing_total_wall_s — 总耗时。若显著高于前两者之和，说明 ROS2 message 构造和 Service 序列化构成不可忽略的开销。
  • timing_avg_per_pose_s — 平均每锚点 IK 耗时，用于评估求解器单次性能基线。

这种分层计时模式使得优化工作可以精准定位到具体环节，而非笼统地"求解慢"。

4.3 排障顺序的依赖分析

9 步最小排障顺序的依赖逻辑：

  1. 构建层（步骤 1-2）：colcon build 通过且 source install/setup.bash 已执行 — 这是 ROS2 节点可运行的前提。
  2. 自检层（步骤 3）：rtfg_geometry_check 通过 — 确认 URDF 解析、KDL Chain 构建、FCL Geometry 加载均正常。
  3. 节点层（步骤 4）：Solver 节点能启动 — 排除动态链接或 URDF 路径问题。
  4. Service 层（步骤 5-6）：load_config 和 fit_preview 能正常响应 — 确认 solver 核心逻辑可用。
  5. 决策层（步骤 7）：判断是否仅为碰撞导致的 success=false — 区分算法问题与参数问题。
  6. 执行层（步骤 8-9）：Controller 激活且 RViz topic 有数据 — 确认执行链路完整。

按此顺序排查，可以避免在 Colcon 未通过或环境未 source 的情况下，浪费数小时分析 IK 算法的收敛性。

5. 结论与改进效果

5.1 接口规范化带来的改进

  • 字段语义明确：每个字段有唯一的语义解释，调用端无需阅读 C++ 源码即可理解参数含义。
  • 数据源单一化：initial_q 来自 URDF，消除了 YAML-URDF 两份数据不一致的隐患。
  • Namespace 隔离：Controller Manager 放入 /rtfg namespace，避免与已有仿真/实车 controller 冲突，降低多系统集成时的运维成本。

5.2 可观测性提升

  • 4 个可视化 Topic（/rtfg/target_tcp_poses、/rtfg/tcp_path、/rtfg/collision_markers、/rtfg/metrics）覆盖了从几何目标到碰撞检测再到性能指标的完整观测链路。
  • 4 个 timing 字段支持精细化的性能瓶颈定位，将优化从"黑盒猜测"升级为"数据驱动"。
  • 失败诊断路径也返回 trajectory，保障了调试可见性。

5.3 排障效率提升

  • 9 步最小排障顺序按依赖关系分层，确保在检查高级问题前所有低级依赖已满足。
  • 常见问题对照表覆盖了从构建、部署到运行时 7 种已知故障模式，每种给出了具体的检查命令和修复方法。
  • timing 字段的分解为性能优化提供了明确的方向：IK 迭代、碰撞检测、序列化三个维度可独立分析和优化。
