5. C++ 性能基准与优化记录

MATLAB MEX vs ROS2 C++ vs 实时架构 性能对照、瓶颈分析、优化路线

5.1 研究背景与性能目标

将轨迹拟合从 MATLAB 环境迁移至 ROS2 C++ 的根本目标不是简单地将 MATLAB 代码翻译为另一种语言，而是为后续高频在线拟合、实时碰撞检测和闭环执行奠定架构基础。具体而言，ROS2 C++ 架构在以下方面具备理论性能优势：

  • 无 MATLAB 解释器开销：移除 MATLAB 运行时解释、GUI 事件循环和 mxArray 类型检查的额外损耗。
  • 编译型计算：运动学、雅可比矩阵、碰撞检测全部在原生 C++ 编译代码中完成，可充分发挥编译器优化（-O3、LTO、向量化）。
  • 原生 FCL 集成：FCL（Flexible Collision Library）以编译库形式提供距离查询和 penetration 检测，无需通过 MATLAB 调用包装层。
  • 常驻节点架构：ROS2 节点可长期驻留，URDF 模型和 FCL 碰撞几何仅加载一次，后续服务调用复用全部计算资源。
  • 并行与缓存潜力：后续可对 IK 求解、碰撞检测和候选搜索进行多线程并行化或结果缓存。
核心性能目标：在复刻 MATLAB/MEX 全部功能和行为的前提下，通过架构升级实现同等或更优的计算性能，并为实时在线执行提供可扩展的底层框架。

5.2 算法与架构设计

5.2.1 搜索策略设计

ROS2 C++ 第一版为了稳定复刻 MATLAB 行为并避免 MEX 版本中"第一锚点失败"的经典问题，采用了更为保守的多阶段搜索策略：

  • 多阶段姿态权重：对每个锚点按轨迹段特性分配四组权重组合（位置、姿态），优先保证末端精度。
  • 三种种子策略：局部种子（基于前一锚点 q_prev 的 ±2π 关节包裹组合）、全局随机种子（97 个固定 rng=42 均匀分布种子）、home/zero/wrist wrap 特殊种子。
  • 碰撞门控：仅对通过 FCL 碰撞距离检查（clearance ≥ 2 mm）的 IK 候选予以接受，未通过则回退至下一组种子。

5.2.3 已验证但未采用的优化（KDL LMA Fast Path）

曾尝试将 KDL LMA（Levenberg-Marquardt 算法）作为每锚点的 q_prev 快速求解路径，但实测结果表明该方案在默认轨迹上反而更慢：

  • KDL LMA 需逐锚点构造 solver 实例，构造开销不可忽略。
  • 在此目标姿态序列上，LMA 不能稳定快速收敛，部分锚点需要多次重试。
  • 整体耗时从约 17-18 s 增加至约 27 s。

因此第一版不默认启用 KDL fast path，主链路保留 DLS（阻尼最小二乘法）求解器。

5.4 结果分析与瓶颈定位

5.4.1 未超越 MEX 基线的根本原因

搜索策略更保守

为避免 MATLAB MEX 中"第一锚点失败"的经典问题，ROS2 C++ 第一版保留了更宽的 fallback 搜索空间：

  • 多阶段姿态权重（四组权重组合逐级尝试）。
  • home / zero / wrist wrap 特殊种子策略。
  • 固定随机全局种子（97 个，rng=42 确保可复现）。
  • 碰撞门控后才接受候选（clearance ≥ 2 mm）。

这提高了轨迹可解释性和连续成功率，但代价是每个 anchor 的候选尝试次数显著增加。

默认轨迹本身存在碰撞

默认 YAML 配置下，playback 审计发现 tool-basin 区域存在穿透（penetration）。对此类参数，求解器在候选评估和回放审计阶段需执行更多的碰撞检测查询，且最终不能缓存为可执行成功轨迹。第二阶段 safe 工况已证明：在 MATLAB 同模型口径下达到 success=true 后，耗时可下降约 20-25%。后续应以 safe 工况作为性能主跑道。

ROS2 响应体更大

FitPreview 服务响应包含以下完整数据：

  • JointTrajectory（关节轨迹）
  • 目标 TCP PoseArray
  • 实际 TCP PoseArray
  • 碰撞点 PoseArray
  • 多个 metrics 字段

同时还将上述数据通过 topic 发布。相比 MATLAB 内部 MEX 返回 struct 指针，ROS2 需要完整的 message 构造、序列化和反序列化。命令行 service call 如打印完整响应，开销更为明显。

碰撞检测仍存在保守审计

第一版已缓存机器人 FCL 几何对象并实现稀疏 playback 检查，但仍需：

  • 频繁更新碰撞对象的 transform（每更新一次调用 setTransform + AABB 计算）。
  • 对每个检查点执行 O(N²) 自碰撞距离查询（跳过 |Δidx| ≤ 1 的相邻链节）。
  • tool-basin 检测：每个 tool link 与多个 basin box 分别计算距离。

5.5 结论与改进效果

5.5.1 当前效果总结

ROS2 C++ 第一版在架构层面完成了以下关键里程碑：

  • 脱 MATLAB 完成：核心 IK 求解、碰撞检测、轨迹回放逻辑已完全脱离 MATLAB 运行时，成为独立 C++ library（rtfg_core）。
  • FCL/URDF 常驻：模型加载和碰撞几何构建在节点启动时一次性完成，服务调用零重复加载开销。
  • 服务化接口就绪：通过三个 ROS2 service（load_config、fit_preview、execute_cached）提供完整功能，可与上游调度系统集成。
  • 可测基线建立：benchmark 框架和验收流程已标准化，可量化追踪每次性能变更。

但需如实记录：当前默认工况的总耗时（约 17-18 s）尚未超越 MEX 历史基线（约 6.9 s）。这是因为第一版的设计优先级为：

行为复刻 > 碰撞可解释性 > 服务化接口 > 代码可维护性 > 绝对性能

性能优化被有意推迟至架构稳定后的第二阶段。

5.5.3 后续性能提升路线

要真正发挥 C++ 编译型计算的优势，下一阶段应集中处理以下瓶颈：

保持 solver 对象和矩阵缓存

当前每个候选内部仍有大量临时对象创建。可将 6×6 雅可比矩阵、H 矩阵、梯度 g、dq、FK 中间值等放入 per-thread workspace，减少 Eigen 的动态内存分配。

引入解析 UR10 IK 或 TRAC-IK

UR10 存在解析 IK 闭式解的可能性。若先生成一组解析候选，再用连续性和碰撞条件筛选，速度将显著优于 DLS 的多轮数值迭代。TRAC-IK 作为替代后端也需要验证，重点测试其对 sensor_shovel_tcp 固定工具偏置和姿态放松策略的兼容性。

将 clearance 纳入优化目标

当前碰撞是 gate（门限），而非 cost（代价）。候选一旦低于门限直接被丢弃，可能导致大量种子重试。若将近壁距离作为连续 cost 加入优化目标函数，可减少因"刚好低于门限"导致的失败回退次数。

分离轻量响应和完整轨迹

FitPreview 可增加 return_full_path 参数，或仅通过 topic 发布完整轨迹路径。服务响应仅返回摘要和轨迹 ID，可降低 CLI 和远程调用的序列化成本。

局部失败重规划

当前若某段发生碰撞，整条轨迹仍按原始几何重新生成。更高效的策略是定位碰撞段后，仅在局部调整 x_plane、姿态权重或 lift 过渡参数，不重新搜索全部 anchor 点。
