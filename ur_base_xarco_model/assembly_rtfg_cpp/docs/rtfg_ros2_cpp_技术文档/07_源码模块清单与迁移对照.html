7. 源码架构与模块映射

MATLAB → ROS2 C++ 逐文件映射表、新增架构文件说明、关键模块职责

7.1 研究背景与目标

在将 MATLAB 原型（realtime_trajectory_fit_gui）迁移至 ROS2 C++ 独立包（assembly_rtfg_cpp）的过程中，核心挑战并非逐行翻译代码，而是在保留算法功能的前提下，按 ROS2 生态的工程惯例进行架构重组。具体需要解决以下问题：

  • 运行时解耦：MATLAB GUI 状态和按钮回调绑定在单一解释器进程中，ROS2 下需拆分为松耦合的 Service / Action 接口。
  • 数据协议标准化：MATLAB struct 和 mxArray 内部格式需替换为 ROS2 .msg/.srv 及 C++ 原生类型（Eigen、标准容器）。
  • 算法层与 ROS 边界层分离：MEX 网关（rtfg_solver_mex.cpp）中同时包含业务流程和数据转换，ROS2 下需将算法逻辑封装为可测试的 ament library，与节点代码分离。
  • 可视化与仿真：MATLAB plot/show 图形迁移为 RViz topic 和 MarkerArray 发布。
  • 基准测试路径：性能评估不再依赖 MATLAB profiler，改为通过 ROS2 service 调用真实节点，输出 benchmark JSON。
迁移总原则：算法逻辑不再绑定 MATLAB 运行时，后续可以被 launch、RViz、Python 客户端、任务规划器或真实控制器复用。GUI 状态和按钮回调迁移为 ROS2 Service；MATLAB 脚本式轨迹计算迁移为可测试 C++ 函数；MEX 内部 C++ 求解器迁移为普通 ament library；MATLAB 绘图迁移为 RViz topic；MATLAB 内部 struct 结果迁移为 ROS2 message 和 benchmark JSON。

7.2.4 关键模块职责

types.h — 内部数据协议

types.h 是新包的内部数据协议，对应 MATLAB state struct 和 MEX struct 的结合体，但更适合 C++ 编译期检查。主要结构体：

  • BasinBox：料斗包络盒，包含名称、尺寸和位姿。
  • SegmentSpec：URDF 运动链段，包含 parent/child、origin、axis、limit。
  • LinkCollision：link 碰撞几何，包含 FCL geometry、link 名称、是否 tool。
  • RobotModel：完整机器人模型，包含 segments、collisions、KDL chain、TCP 偏置。
  • CandidateInfo：单点 IK 候选，包含 q、位置误差、姿态误差、clearance、cost。
  • CollisionSummary：单配置碰撞摘要。
  • SolverConfig：clearance、IK 容差、最大迭代、阻尼。
  • TrajectoryResult：整条轨迹结果，包含 anchor、playback、TCP path、碰撞点、metrics。

utils.cpp — 数学工具函数

提供数学基础函数，其中 poseError 是 IK 的核心误差定义，把位置误差和旋转对数误差组合成 6 维向量。quinticBlend 同时被姿态过渡和 playback 插值使用，保证端点速度和加速度连续。

函数用途
rad2deg弧度 → 度
rtToTform旋转矩阵 + 平移 → 4×4 齐次矩阵
rpyToRotRPY → 旋转矩阵
xyzrpyToTform位置 + RPY → 齐次矩阵
rotToLogVec旋转矩阵 → 对数向量（旋转向量）
rotationDistance两姿态间旋转角度距离
quinticBlend五次多项式混合插值
slerpRotation球面线性插值旋转
wrapJointDelta关节增量包裹到 ±2π
continuityCost连续性代价计算
poseError6 维位姿误差（位置误差 + 旋转对数误差）

robot_model.cpp — 机器人模型加载

负责把 URDF 变成求解器可直接使用的模型：

  1. 读取 solver URDF。
  2. 从 tip 反向找到 base。
  3. 正向生成 SegmentSpec。
  4. 读取 joint limit。
  5. 加载 link collision geometry。
  6. 构建 KDL chain。
  7. 计算 ur10_wrist_3 → sensor_shovel_tcp 固定偏置。

mesh 处理保留在 C++ 层。STL 被读取为 FCL BVH 模型，避免每次服务调用重新解析。package:// 路径在 solver URDF 中被改为相对路径，是为了让普通文件读取逻辑更简单、更稳定。

trajectory_generator.cpp — 轨迹生成

迁移了 MATLAB 轨迹生成主线：

  • YAML 加载。
  • 参数限幅。
  • 环境位姿构造。
  • YOZ 三段点列。
  • 2D 到 3D。
  • TCP 姿态生成。
  • 料斗包络盒生成。
  • 初始关节角从 URDF ros2_control block 中读取。
设计要点：初始关节角不是写死在 C++，而是从 URDF 的 <state_interface name="position"><param name="initial_value">...</param> 中解析。这样 URDF 初始姿态变化时，服务默认 current_q 会自动跟随。

ik_solver.cpp — IK 求解器

IK 求解器保留 MATLAB MEX 的 DLS 思路，但去掉 MATLAB 依赖：

函数算法关键参数
solveSinglePose阻尼最小二乘法 (DLS)λ 自适应 [5e-4, 0.1]，迭代 ≤ 60，停滞检测 8 步
buildSeedList3 基种子 × 9 包裹组合 (J5, J6 ±2π)去重容差 1e-9 rad
buildGlobalSeedList中位 + 96 均匀随机 (rng=42)关节限位内均匀分布
solveSinglePoseKdlKDL LMA 后端（当前非默认快路径）保留为后续可选后端

trajectory_solver.cpp — 轨迹拟合编排器

这是拟合编排器，对应 MATLAB/MEX 中最重要的逻辑，也是后续性能优化的主战场。当前耗时主要集中在这里，而不是 ROS2 launch 或 RViz。

  1. 遍历目标 TCP。
  2. 对每个目标按权重阶段和 seed 阶段求 IK。
  3. 对达标候选做延迟碰撞检测。
  4. 按连续性代价选择最优候选。
  5. 保存 anchor 关节角。
  6. 生成 playback 轨迹。
  7. 稀疏审计碰撞。
  8. 汇总 metrics。

collision_checker.cpp — 碰撞检测

检测类型方法备注
自碰撞 (self)O(N²) 对距离查询，跳过相邻链节 (|Δidx| ≤ 1)第一版已避免重复构建 FCL geometry
工具-本体 (tool-body)工具链节 × 本体链节距离查询缓存料斗 collision object
工具-料斗 (tool-basin)工具链节 × 料斗 Box 距离查询每次评估主要更新 transform 和执行 FCL distance

rtfg_solver_node.cpp — ROS2 节点边界层

ROS2 节点只做"ROS2 边界层"工作，不包含具体算法：

  • 创建 service。
  • 创建 publisher。
  • 创建 action client。
  • 加载默认资源路径。
  • 把 C++ 内部结果转换为 ROS2 message。
  • 缓存可执行轨迹。
  • 处理异常并返回 service response。
架构要点：以后如果增加 GUI 或真实控制器，核心算法仍在 rtfg_core（算法核心层），不会被节点代码绑死。

7.4 结果分析

7.4.1 模块职责边界清晰度提升

迁移前，MEX 网关 rtfg_solver_mex.cpp 同时承担数据转换、错误处理、算法调度三种职责。迁移后：

  • ROS2 边界层（rtfg_solver_node.cpp）：只处理 Service 生命周期、message 转换、异常捕获回传。
  • 算法核心层（trajectory_solver.cpp 等 6 个模块）：完全不感知 ROS2 数据类型，只操作 Eigen/KDL/FCL 原生类型。
  • 数据协议层（types.h）：统一内部结构体定义，MATLAB struct 和 MEX struct 合并为编译期检查的类型。

7.4.2 可测试性与复用性分析

  • 算法核心层编译为 ament library，可在不启动 ROS2 节点的前提下进行单元测试。
  • 三个 Service 定义（LoadConfig、FitPreview、ExecuteCached）构成完整的公共操作面，Python 客户端、任务规划器、GUI 均可复用同一接口。
  • Benchmark 通过 ROS2 service 调用真实节点，测得端到端性能而非模块级微基准，更贴近实际运行体验。
  • 初始关节角从 URDF ros2_control block 中解析，URDF 变化时自动跟随，无需修改 C++ 代码。

7.4.3 潜在瓶颈分析

  • trajectory_solver.cpp 包含遍历目标 TCP、多权重阶段、多 seed 阶段、延迟碰撞检测、最优候选选择、playback 生成、稀疏审计等全部编排逻辑，是当前性能最集中的模块。
  • collision_checker.cpp 虽然已缓存 FCL geometry 和料斗 collision object，但 O(N²) 自碰撞对查询在密集场景下仍有优化空间。
  • ik_solver.cpp 的 KDL LMA 后端经测试比 DLS 更慢，不作为默认快路径，但保留为后续可选后端。
