4. 分层碰撞检测流水线与狭小姿态策略

CollisionPipeline 设计（粗筛 + FCL 精检）、三类碰撞检测、2 mm clearance 策略、自适应 stride

4.1 研究背景与目标

4.1.1 碰撞检测的双重使命

第一版碰撞检测服务于两个目的：

  1. IK 候选阶段过滤：在逆运动学求解过程中，对收敛的候选解进行碰撞安全检查，过滤掉明显不安全的候选，避免无效解进入后续轨迹生成流程。
  2. Playback 轨迹审计：在回放轨迹生成完成后，审计整条轨迹的最小 clearance 和碰撞对象，确保最终输出的轨迹在几何上是可执行的。

返回结果不仅告诉用户成功或失败，还要说明失败发生在哪一类对象、哪一段轨迹、哪个 sample 附近。这种细粒度的反馈机制是后续调试和参数调整的关键支撑。

4.1.2 狭小姿态的核心挑战

料斗工况对 UR10 来说存在几个典型难点：

  • 空间约束复杂：工具 tip 要进入料斗内部，空间由壁厚、内宽、泥面高度共同约束，可用工作空间极为有限。
  • 末端偏置放大效应：sensor_shovel_tcp 相对 wrist 有固定偏置，wrist 姿态稍有变化会造成 tip 大幅扫动，微小的关节调整即可引发显著的 TCP 位移。
  • 姿态突变集中：圆弧到出泥阶段姿态变化集中，wrist 关节容易接近奇异或发生等效角跳变，导致 IK 求解不稳定。
  • 目标冲突：严格姿态目标可能和碰撞 clearance 冲突，单纯追求位姿精度可能导致穿模，而过度保守的 clearance 又可能使 IK 无解。

因此，算法不能只做"每点 IK 最小误差"，而要把位置、姿态、连续性、clearance 作为一个综合选择问题，在多个目标之间寻求平衡。

4.2 算法与架构设计

4.2.1 分层碰撞检测流水线

CollisionPipeline 采用"粗筛 + 精检"的分层设计理念，在计算成本和检测精度之间取得平衡：

4.2.2 FCL 模型来源

机器人碰撞几何来自 assembly_rtfg_solver.urdf：

  • 基本几何：box/cylinder 直接转换为 FCL 基本几何类型。
  • Mesh 模型：STL mesh 读取为 fcl::BVHModel<fcl::OBBRSSd>，构建 BVH 树加速距离查询。
  • 工具标识：tool link 标记为 sensor_shovel 和 sensor_shovel_tcp，用于区分工具与本体。

料斗碰撞几何来自算法层包络盒，而不是 mesh 三角面。包络盒包括左右壁、前后壁、底部等对象，名称会进入 collision result。这样做有两个理由：

  • 稳定性：对狭小空间 clearance 计算，规则 box 比高面数 mesh 更稳定，不会因三角面离散化引入数值噪声。
  • 可解释性：业务上更关心"左壁/右壁/底部"这样的语义对象，而不是 STL 三角面 ID，语义化名称便于调试和定位问题。

4.2.3 三类碰撞检测

CollisionSummary 将碰撞检测划分为三类最小距离：

字段 含义 优先级
min_self 机器人非相邻 link 间自碰撞距离 3（最低）
min_tool_body 工具与机器人本体距离 2
min_tool_basin 工具与料斗包络距离 1（最高）

违规优先级为：tool_basin > tool_body > self。因为当前任务的主要风险是铲尖或工具进入料斗狭小区域时打到料斗壁，其次是工具碰本体，最后才是非相邻 link 自碰。这种优先级排序直接影响碰撞违规时的消息组织和排查顺序。

4.2.4 2 mm Clearance 门限

默认碰撞门限：

clearance_threshold = 0.002 m

候选配置的最小 clearance 小于该值即视为违规。FCL 返回负值表示 penetration（穿透）。文档和 metrics 中出现 -1.0 时表示 FCL 识别到穿透，而不是普通正距离。

设计取舍：2 mm 是第一版保守门限。它足够小，可以允许狭小工况贴近料斗壁；又足够明确，可以把穿模和贴壁风险显式暴露出来，为后续调参提供明确的量化依据。

4.2.5 自碰撞跳过规则

URDF 中 base_jizuo、base_jizuo_base_ur10_with_dizuo 和 ur10 是固定安装结构，几何上可能天然接触或嵌套。如果直接按普通非相邻 link 检测，会出现 base_jizuo <-> ur10 这类假阳性，即几何上本应存在的安装接触被误报为碰撞。

第一版显式跳过这些安装基座之间的 pair：

base_jizuo
base_jizuo_base_ur10_with_dizuo
ur10

这个规则只适用于固定安装基座，不跳过工具、腕部或料斗相关碰撞，确保安全关键区域始终处于检测范围内。

4.2.6 候选级碰撞门控

IK 候选阶段的碰撞检查采用 延迟触发 策略，避免对明显未收敛的候选做 FCL 计算。流程如下：

solve IK candidate
if position/orientation error acceptable:
  run FCL distance
  if clearance >= threshold:
    candidate is safe

相比 MATLAB GUI 中常见的"先生成再全量检查"方式，这种延迟门控方式可以更早过滤不安全解，减少后续回放轨迹的风险，同时降低无效 FCL 计算的开销。

4.2.7 Playback 稀疏审计与自适应 Stride

回放轨迹点数通常是 anchor 的 4 到 6 倍。每个 playback 点都完整 FCL 检查会明显增加耗时，因此第一版采用 稀疏审计 策略：

collision_stride = 10

审计规则：始终检查 第一点、最后一点，以及每 10 个 playback 点。由于插值阶段已经限制关节步长、TCP 位置步长和姿态步长，相邻点之间不应出现大幅跳变，稀疏采样足以暴露绝大多数碰撞风险。

注意：稀疏审计不是最终工业安全策略。当前版本作为第一版实现，优先保障求解效率。后续如果要真实执行，应在执行前增加全量碰撞审计或控制器侧实时监控，以实现完整的安全保障。

4.2.8 六层综合解决策略

针对狭小姿态的复杂性，第一版组合使用以下六层策略：

策略层级 具体措施 解决的问题
轨迹层 圆弧末端到出泥加入 slerp 姿态过渡 降低姿态突变，减少 wrist 奇异风险
IK 层 四阶段姿态权重调度，优先位置和安全 在姿态精度不足时优先保证位置收敛和安全性
种子层 使用上一点、home、zero、wrist 等效角和固定随机种子 增加找到可行解的覆盖范围
连续性层 用 q_prev + dq_prev 作为参考，压制角度跳变 保证相邻锚点间关节角度平滑过渡
碰撞层 候选级延迟 FCL 门控 + playback 稀疏审计 在可接受计算成本内过滤碰撞风险
回传层 失败时返回碰撞对象、段名、sample index 和最小 clearance 提供细粒度调试信息，便于定位问题

4.4 结果分析

4.4.1 默认参数下的碰撞行为

在默认 YAML 配置下，IK 可以完成求解，但 playback 碰撞审计发现 tool-basin penetration。服务返回：

success=false
message="trajectory solved but collision audit found violations"

这一结果包含以下重要信息：

  • 轨迹求解器没有崩溃：IK 求解和轨迹插值模块运行正常，数值上稳定。
  • 关节轨迹和 TCP path 已生成：轨迹数据已经完整产生，可进一步分析。
  • 碰撞检测认为该参数下的轨迹不应直接执行：安全门限生效，阻止了潜在的危险轨迹。

4.4.2 调参方向

用户需要调整以下参数以消除碰撞违规：

  • left_wall_offset — 料斗左壁偏移量
  • x_plane — 切入平面位置
  • theta_deg — 铲尖角度
  • depth — 挖掘深度
  • 环境位姿（料斗相对于机器人的全局位姿）

或者继续增加避障策略，如局部轨迹调整或 clearance 约束加入 IK cost。

4.4.3 瓶颈定位

当前版本的主要瓶颈和限制：

  • 碰撞检测仅在候选过滤阶段发挥作用，不作为 IK 求解代价函数的一部分，因此无法引导求解器主动远离障碍物。
  • 稀疏审计存在漏检风险，stride=10 在极端情况下可能遗漏单点穿模。
  • 料斗包络盒的精度依赖手写常量，如果包络盒与实际 mesh 偏差过大，可能出现误报或漏报。

4.5 结论与改进效果

4.5.1 当前效果总结

第一版碰撞检测模块已实现以下核心能力：

  • 基于 FCL 的精确距离查询，支持 BVH mesh 和基本几何。
  • 三类碰撞检测全覆盖（自碰撞、工具-本体、工具-料斗），附带语义优先级。
  • 候选级延迟 FCL 门控，避免无效计算。
  • Playback 稀疏审计，在计算成本和检测覆盖率之间取得平衡。
  • 细粒度碰撞信息回传（碰撞对象、段名、sample index、最小 clearance）。
  • 可配置的 2 mm clearance 门限。

4.5.2 后续改进方向

为了让复杂狭小姿态真正稳定，需要继续做三类增强：

增强类别 具体措施 预期收益
几何增强 从 URDF/mesh 自动提取料斗内壁精确包络，减少手写 box 常量和 mesh 之间的不一致 消除包络误差导致的误报/漏报
优化增强 把 collision clearance 加入 IK cost，而不是只做候选过滤 引导求解器主动远离障碍物，提高狭小空间求解成功率
搜索增强 对碰撞段局部回退，自动调整姿态权重、x_plane 或局部 lift 轨迹，而不是整条轨迹失败 从"全有或全无"变为"局部修复"，提高求解鲁棒性
架构升级价值：C++ 实现的 CollisionPipeline 相比 MATLAB 原型在可扩展性、实时性和代码可维护性方面均有显著提升。分层设计使得后续增加新的碰撞检测类型（如动态障碍物、环境地图）只需扩展检测层，无需改动 IK 和轨迹生成核心逻辑。候选级门控和稀疏审计为实时在线碰撞检测奠定了性能基础。
