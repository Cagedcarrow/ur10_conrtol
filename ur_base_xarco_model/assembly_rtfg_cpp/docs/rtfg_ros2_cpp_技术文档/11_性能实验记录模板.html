11. 性能实验记录模板与对比口径

实验记录表模板、测试环境说明、对比口径、复现步骤、优化有效性判断规则

核心原则：轨迹拟合性能很容易被不同条件混淆。一次 service call 的总耗时不仅取决于 C++ IK，还取决于轨迹点数、当前关节角、是否有碰撞、clearance 门限、是否发布完整 PoseArray、是否打印完整 service response、是否启动 RViz 和 MoveIt、编译类型是 Debug 还是 Release、CPU 当时是否被其他 ROS 节点占用。因此，后续每次说"快了多少"都必须记录完整实验条件，否则不能判断是算法真的变快，还是点数减少、碰撞减少或输出变少。

1. 研究背景与目标

1.1 为什么需要标准化实验记录

在 ROS2 C++ 迁移过程中，性能对比面临多重混淆因素：

  • 轨迹点数变化：不同的 anchor_count 和 playback_count 直接影响总耗时，不能跨点数直接比较耗时绝对值。
  • 碰撞状态差异：有碰撞 vs 无碰撞路径的 IK 求解路径不同，碰撞检测耗时也不同。
  • 输出负载差异：是否发布完整 PoseArray、是否打印完整 service response 消息体，会影响 wall clock 计时。
  • 运行环境差异：Debug vs Release 编译、RViz/MoveIt 是否启动、CPU 负载状态，都会引入显著偏差。

因此，建立标准化的实验记录模板是对比口径一致性的基础，也是后续判断优化是否有效的必要条件。

4. 结果分析

4.1 当前瓶颈分析

从默认工况数据可以定位到以下瓶颈：

  • IK 总耗时约 15 s：占 total_wall_s（~17-18 s）的 83-88%，是主要瓶颈。原因是第一版保留了保守的多阶段 DLS、多种子搜索和候选级碰撞门控策略。
  • 碰撞检测约 2.6-2.8 s：占总量约 15%，包含自碰撞、工具-本体、工具-料斗三个维度的 FCL 距离查询。
  • ROS2 序列化和 topic 发布：包含 PoseArray、MarkerArray、DisplayTrajectory 等可视化数据的构造和发布，额外增加 ~0.5-1.0 s 开销。

4.2 优化有效性判断规则

一次优化只有在以下所有条件满足时才可以说有效：

  • 相同 YAML。
  • 相同 current_q。
  • 相同 clearance threshold。
  • 相同 build type。
  • anchor_count 和 playback_count 基本一致。
  • success/collision 状态一致，或明确解释变化。
  • 至少重复 3 次，取中位数。
重要：如果优化同时改变了轨迹点数或碰撞状态，应分开解释，不能只用总耗时下降证明算法变快。

5. 结论与改进效果

5.1 当前结论

  • 第一版完成 ROS2 C++ 迁移和可测基线。
  • 当前默认工况还没有超过历史 MEX 基线。
  • 未提升的原因是保守搜索策略、碰撞诊断、ROS2 message 回传和默认参数碰撞。
  • C++ 的优势在于后续可以常驻模型、缓存对象、解析 IK、并行碰撞和轻量化响应。
文档表述要求：任何对外文档都应保持以上说法。不要写成"ROS2 C++ 已经大幅提升"，因为当前实测不支持这个结论。

5.2 架构升级方向

当前 ROS2 C++ 第一版虽然性能尚未超过 MEX 基线，但其架构设计为后续优化奠定了基础：

  • 常驻模型：C++ 节点在生命周期内保持 RobotModel 实例，避免重复 URDF 解析和 KDL/FCL 对象构建。
  • 对象缓存：FCL CollisionObject 使用 thread_local 缓存，后续调用仅更新变换矩阵和 AABB，避免 BVH 树重建。
  • 解析 IK 潜力：可引入 UR10 专用解析 IK（如 TRAC-IK 或 KDL IKFast），替代当前纯数值 DLS 求解器。
  • 并行碰撞：碰撞检测可并行化，利用多核 CPU 加速 O(N^2) 自碰撞查询。
  • 轻量化响应：可配置是否发布 PoseArray、MarkerArray 等可视化数据，降低序列化开销。

5.3 后续记录规范

每次性能实验后，按以下步骤操作：

  1. 确认实验条件与上一版一致（YAML、current_q、threshold、build type）。
  2. 使用结构化模板记录所有字段。
  3. 至少重复 3 次，取中位数作为最终数值。
  4. 保存 JSON 到 benchmarks/results/ 目录。
  5. 在与 MEX 对比时明确标注使用的是 Core Benchmark 还是 Service Benchmark。
  6. 结论中区分"算法提升"和"条件变化"带来的性能差异。
