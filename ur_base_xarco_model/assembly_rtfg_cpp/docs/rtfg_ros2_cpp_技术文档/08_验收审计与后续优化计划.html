8. 验收结果与后续路线图

对用户目标逐条验收、当前缺口分析、P0-P4 优先级的后续优化路线

8.1 研究背景与目标

将 MATLAB realtime_trajectory_fit_gui 的轨迹生成、运动学拟合、碰撞审计和可视化能力迁移到 ROS2 Humble C++ 环境，不是简单的语言替换，而是要在摆脱 MATLAB 运行时的同时，为后续高频在线拟合、实时碰撞检测和真机控制执行提供可扩展的工程基础。

用户原始目标可以拆解为十项验收指标，覆盖功能包创建、模型复用、算法复刻、性能基线、文档完整性和执行入口等维度。本章对每一项进行对照审计，识别当前第一版的实际达成状态与未完全达成项，并据此制定下一阶段的优化路线图。

8.1.1 用户目标拆解

  1. 新建功能包 — 在当前 ROS2 工作区新建功能包，不替换现有 assembly_rviz_tcp_control。
  2. URDF 复用 — 使用 MATLAB 版本代码中的 URDF。
  3. 轨迹生成复刻 — 复刻 realtime_trajectory_fit_gui 的轨迹生成功能。
  4. 运动学复刻 — 复刻尖端点拟合和运动学功能。
  5. C++ 实现 — 尽量用 C++ 实现，摆脱 MATLAB/MEX 网关。
  6. 性能基线 — 提供高性能拟合和碰撞检测基础。
  7. ROS2 入口 — 提供 ROS2 运行入口、可视化和执行入口。
  8. 中文文档 — 编写中文技术文档，内容不少于 MATLAB 技术文档。
  9. 性能说明 — 文档必须说明性能提升多少、原因是什么。
  10. 算法说明 — 文档必须说明轨迹生成逻辑、轨迹拟合原理、复杂狭小姿态策略和 ROS2 节点图。

这十项验收指标共同定义了从 MATLAB 原型到 ROS2 C++ 可部署系统需要满足的完整功能面。

8.2 验收框架与方法

验收采用三层验证方法：

  • 功能验收 — 逐项检查功能是否可用，通过 service call、launch 和 benchmark 脚本验证。
  • 性能验收 — 通过 rtfg_benchmark.py 记录总耗时、IK 耗时、碰撞检测耗时和加速比，与 MATLAB/MEX 历史基线横向对比。
  • 完整性验收 — 检查文档覆盖面和算法说明是否达到或超过 MATLAB 技术文档的深度。

8.3.2.1 默认参数无碰撞执行

8.4 结果分析与瓶颈定位

8.4.1 为什么当前没有超过 MEX

搜索策略更保守

为了避免 MATLAB MEX 中"第一锚点失败"一类问题，ROS2 第一版保留了更宽的 fallback 搜索：多阶段姿态权重、home/zero/wrist wrap 种子、固定随机全局种子、碰撞门控后才接受候选。这提高了可解释性和连续轨迹成功率，但会增加每个 anchor 的候选尝试次数。

默认轨迹本身有碰撞

默认 YAML 下 playback 审计发现 tool-basin penetration。对此类参数，求解器会在候选和审计阶段做更多碰撞判断，且最终不能缓存为可执行成功轨迹。当前 benchmark 测到的是"带碰撞风险的诊断路径"，而非"理想无碰撞轨迹"的最快路径。第二阶段 safe 工况已证明：在 MATLAB 同模型口径下达到 success=true 后，耗时可显著下降。

ROS2 响应体更大

FitPreview 返回 JointTrajectory、目标 TCP PoseArray、实际 TCP PoseArray、碰撞点 PoseArray 及多个 metrics 字段，同时还发布 topic。相比 MATLAB 内部 MEX 返回 struct，ROS2 需要 message 转换和序列化，CLI 远程调用开销更明显。

碰撞检测仍有保守审计

第一版虽已缓存机器人 FCL 几何、对 playback 做稀疏检查，但仍频繁更新碰撞对象 transform、计算 AABB、执行距离查询。tool-basin 包络虽是 box，但每个 tool link 与多个 basin box 都需计算距离。

8.4.2 耗时分布分析

当前 benchmark 数据显示 IK 耗时占总耗时的 85-90%，碰撞检测约占 10-15%，其余为 ROS2 序列化和消息构造。IK 是当前绝对瓶颈。

耗时模块 诊断工况占比 safe 工况占比 优化优先级
IK 求解（DLS 迭代 + 多种子搜索） ~89.9% ~89.9% P1-P2
碰撞检测（FCL 距离查询） ~10.1% ~10.1% P3
ROS2 序列化 / topic 发布 < 1% < 1% P4

8.4.3 当前实现的价值

虽然性能目标尚未完全达成，但当前包已将关键障碍拆开：

  • MATLAB 依赖已经解除。
  • URDF 和 meshes 已复制到 ROS2 包。
  • 算法可以通过 colcon build 编译。
  • 节点可以常驻并反复服务调用。
  • benchmark 可以重复测量。
  • 碰撞失败可以解释为具体对象和段名。
  • 后续优化有了可量化基线。
基线的重要性：没有这个基线，后续很难判断"改快了多少"或"碰撞是不是被误报"。当前第一版满足"可运行迁移基线"，但还不满足"性能大幅提升且默认无碰撞执行"的最终目标。

8.5 结论与后续路线图

8.5.1 当前效果总结

第一版已完成以下核心架构升级：

  • 脱 MATLAB：核心算法作为普通 C++ library rtfg_core 存在，不再依赖 MATLAB 运行时。
  • ROS2 服务化：算法逻辑通过三个 service 暴露：LoadConfig、FitPreview、ExecuteCached。
  • 模型常驻：URDF 解析、FCL 碰撞几何在节点启动时一次性加载，避免每次服务调用重复初始化。
  • 可视化接入：通过 RViz PoseArray、MarkerArray 和 MoveIt DisplayTrajectory 展示结果。
  • 可测基线：benchmark 脚本可重复测量并记录性能数字。

当前文档覆盖 11 个 HTML 文件，内容涵盖架构、轨迹生成、IK 算法、碰撞检测、性能基准、接口规范、模块清单、验收审计、调参手册和排障指南，深度和广度均超过原 MATLAB 技术文档。

P0（最高优先级）：得到无碰撞默认工况

先不要急着优化速度。当前默认参数有 tool-basin penetration，性能测到的是失败诊断路径。应先找到一个无碰撞参数集：

  • 增大 left_wall_offset，远离侧壁。
  • 调整 x_plane，避开工具与壁的最近区域。
  • 减小 depth。
  • 减小或调整 theta_deg。
  • 检查 base_jizuo_to_block_with_basin_frame 位姿是否和 URDF mesh 完全一致。

得到无碰撞参数后，保存为第二个 YAML，例如：

config/environment_runtime_config_safe.yaml

然后分别 benchmark：

  • default collision diagnostic。
  • safe execution path。

P1：减少 IK 搜索次数

目前最主要耗时是 IK。优化方向：

  • 统计每个点成功发生在哪个 seed/stage。
  • 对连续成功段只尝试 q_prev 和前一阶段权重。
  • 只有连续失败时才启用 home/zero/global seeds。
  • 缓存上一点成功的姿态权重阶段。
  • 对碰撞段做局部参数调整，而不是全局随机种子。

P2：引入解析 IK

UR10 可以考虑解析 IK：

  1. 先把目标 sensor_shovel_tcp 转换到 ur10_wrist_3。
  2. 求所有 wrist 解。
  3. 按 joint limit、连续性和 clearance 排序。
  4. 对最优候选做一到两步 DLS refine。
预期提升：这条路线最可能带来数量级提升，因为它绕过了每点多轮 DLS 迭代。解析 IK 可从 O(N * M) 降至 O(N + M)，其中 N 为锚点数、M 为每点迭代次数。

P3：碰撞检测并行化

Playback 审计的每个 sample 基本独立。可以用线程池并行：

  • 每个线程持有自己的 FCL collision object workspace。
  • 汇总全局最小 clearance。
  • 汇总第一个碰撞点。
注意：FCL 对象复用需要线程局部存储（TLS），不能多个线程同时改同一个 collision object transform。

P4：响应体轻量化

当前 service 返回完整 trajectory 和 pose array。后续可以增加模式：

preview_level = summary_only | trajectory | full_debug

常规 benchmark 用 summary_only，RViz 通过 topic 接收完整 path。这样可以把 ROS2 序列化从核心算法耗时中分离出来。
