1. ROS2 节点架构与数据流

包结构、ROS2 节点图、数据流、服务/话题接口、求解链组装、与 MATLAB 架构对比

1. 研究背景/目标

MATLAB realtime_trajectory_fit_gui 的核心工作流是：读取环境与轨迹参数，生成料斗局部 YOZ 平面轨迹，将其变换到 UR10 基坐标，构造每个 TCP 目标姿态，使用 C++ MEX 求解连续关节轨迹，最后在 MATLAB GUI 中预览、碰撞检查和回放。该流程依赖 MATLAB 运行时、GUI 回调框架和 MEX 编译链路，无法直接嵌入 ROS2 生态。

ROS2 第一版的目标是将这套算法从 MATLAB/MEX 环境完整迁移到 ROS2 C++ 原生节点，实现以下设计目标：

  • 脱 MATLAB 运行：不再依赖 MATLAB 解释器和 MEX 编译链路，所有计算在 ROS2 节点内完成
  • 服务化接口：将 MATLAB GUI 的按钮回调映射为 ROS2 服务，使后续 GUI、网页前端、Python 客户端或任务调度器都能调用同一套服务
  • RViz/MoveIt 可视化：用 RViz PoseArray、MarkerArray、MoveIt DisplayTrajectory 替代 MATLAB 绘图
  • 独立验证台：不修改现有 assembly_rviz_tcp_control，不覆盖已有 assembly_description 或 assembly_moveit_config，新增独立 ament C++ 包作为迁移阶段的独立验证台
设计边界：第一版故意不做完整 GUI 克隆。迁移优先级是把算法从 MATLAB/MEX 迁到 ROS2 C++，并能用服务、RViz 和 fake controller 验证。后续如果需要 GUI，可以在这个服务层上增加 Qt、rqt、web 或 RViz panel，而不再把轨迹/IK/碰撞逻辑嵌入界面。

2.3 节点运行时职责

rtfg_solver_node 把 MATLAB GUI 中分散在按钮回调、状态结构体和 MEX 调用里的逻辑合并成 ROS2 服务，运行时职责包括：

  • 启动时定位包内默认 YAML、solver URDF 和 mesh 目录
  • 用 urdfdom 解析 URDF，建立 base_jizuo → sensor_shovel_tcp 的运动学链
  • 用 KDL 解析 base_jizuo → ur10_wrist_3 的六轴链，保留固定工具偏置
  • 读取 runtime YAML 中的轨迹参数、环境位姿和 URDF 初始关节角
  • 调用 buildTargetPlan 生成目标 TCP 序列
  • 调用 solveTrajectory 求解关节锚点和回放轨迹
  • 发布目标 TCP、实际 TCP、碰撞 marker、性能 metrics 和 MoveIt display trajectory
  • 缓存最近一次拟合成功且无碰撞违规的 trajectory，供 /rtfg/execute_cached 执行

2.4 数据流

第一版数据流是明确的单向管线，共 8 个阶段：

  1. LoadConfig — 读取 YAML 和 URDF 初始关节角
  2. FitPreview — 接收显式参数；如果字段为空或当前关节角为空，则回退到默认配置
  3. clampTrajectoryParams — 对输入参数做料斗几何约束，防止生成明显越界的几何轨迹
  4. buildTargetPlan — 生成目标点、目标齐次变换和段名
  5. buildBasinCollisionBoxes — 生成料斗 wall/floor 的 FCL 盒模型
  6. solveTrajectory — 逐点 IK，生成 anchor 关节序列
  7. solveTrajectory — 生成 quintic playback 轨迹并进行稀疏碰撞审计
  8. 节点输出 — 把结果转换为 ROS2 message，发布到 RViz/MoveIt topic

2.5 模型边界

新包内包含两份模型文件，分别服务于不同用途：

模型文件用途Mesh 路径
assembly_rtfg.urdf.xacro ROS2 runtime、robot_state_publisher、ros2_control、MoveIt package://assembly_rtfg_cpp/urdf/meshes
assembly_rtfg_solver.urdf C++ solver 直接读取（避免 package URI 解析） 相对路径
SRDF 定义：group assembly_rtfg_manipulator，base 为 base_jizuo，tip 为 sensor_shovel_tcp。该选择与 MATLAB 版尖端点拟合一致，避免用 UR10 wrist link 作为工具端导致轨迹误差。

3. 对比效果

3.1 MATLAB GUI 与 ROS2 服务接口对照

MATLAB GUI 版本通过 action dispatcher 或回调分派不同按钮逻辑。ROS2 第一版没有复制 GUI，而是把这些 action 映射成独立服务：

MATLAB GUI 动作ROS2 服务/Topic说明
读取 runtime YAML /rtfg/load_config 加载配置参数和初始关节角
生成并显示目标轨迹 /rtfg/fit_preview + /rtfg/target_tcp_poses 生成目标 TCP 序列并发布 PoseArray
拟合关节轨迹 /rtfg/fit_preview 返回 JointTrajectory IK 求解 + quintic 回放
显示实际 TCP path /rtfg/tcp_path 发布回放轨迹的 TCP 路径
显示碰撞点 /rtfg/collision_markers MarkerArray 显示碰撞位置和 clearance
查看耗时 /rtfg/metrics 发布各阶段耗时统计
执行最近一次轨迹 /rtfg/execute_cached 将缓存的轨迹发送给 controller

4. 结果分析

4.1 架构分析

从架构角度看，第一版实现了以下关键改进：

  • 关注点分离：MATLAB 版将轨迹生成、IK 求解、碰撞检测、可视化全部耦合在 GUI 回调中。ROS2 版通过 rtfg_core 库将每个阶段封装为独立模块，头文件定义接口，源文件实现逻辑，模块间通过结构体传递数据。
  • 服务化接口：三组 service（LoadConfig、FitPreview、ExecuteCached）对应 MATLAB GUI 的加载配置、拟合预览、执行回放三个核心动作。这个映射保留了旧工作流的操作顺序，同时让任意客户端都能调用。
  • 双模型策略：xacro 用于 ROS2 生态（package:// URI），solver URDF 用于 C++ 求解器（相对路径），避免 urdfdom/FCL 加载时解析 package URI 的开销。
  • 数据流管线化：8 阶段单向数据流明确了每个步骤的输入输出，有利于后续性能分析（逐阶段计时）和并行化（如 IK 求解与碰撞检测管线重叠）。

4.2 当前状态

当前实测状态：第一版已经完成脱 MATLAB、ROS2 服务化和 RViz/MoveIt 可视化接入。默认 YAML 参数可以生成完整关节轨迹、TCP path 和碰撞审计结果。但当前轨迹在 tool-basin 检查中存在穿透，因此 /rtfg/fit_preview 会返回 success=false 并附带碰撞对象、最小 clearance 和轨迹数据。第二阶段已新增 config/environment_runtime_config_safe.yaml 用于无碰撞成功路径验证。
性能说明：当前 ROS2 C++ 第一版还没有超过历史 MEX 基线。主要原因不是 C++ 本身慢，而是第一版为了稳定复刻 MATLAB 行为，保留了更保守的多阶段 DLS、多种子搜索、候选级碰撞门控，以及 ROS2 service/topic 的序列化和可视化数据发布。详细分析见 05_Cpp性能基准与优化记录。

5. 结论与改进效果

第一版 assembly_rtfg_cpp 完成了从 MATLAB/MEX 到 ROS2 C++ 的架构迁移，核心成果包括：

  • 成功脱 MATLAB：算法完全在 ROS2 Humble 原生 C++ 节点中运行，无任何 MATLAB 依赖
  • 模块化重构：原 MATLAB MEX 单文件拆分为 7 组头文件/源文件（types、utils、robot_model、ik_solver、collision_checker、trajectory_solver、trajectory_generator），接口清晰、可单元测试
  • 服务化接口：3 个 ROS2 service 覆盖 MATLAB GUI 的核心操作，为后续多客户端接入奠定基础
  • 可视化迁移：MATLAB 绘图全部替换为 RViz + MoveIt 标准消息，可与现有 ROS2 可视化工具链无缝集成
  • 独立验证台：新包不修改现有 assembly_rviz_tcp_control、assembly_description、assembly_moveit_config，可独立开发测试

待改进方向：

  • 性能优化：当前第一版尚未超越 MEX 基线，需在保持功能一致的前提下做针对性加速（见 05_Cpp性能基准与优化记录）
  • 碰撞参数调优：默认 YAML 在 tool-basin 检查中存在穿透，需进一步调整轨迹参数或碰撞模型（见 04_碰撞检测与狭小姿态策略）
  • GUI 层：后续可在当前服务层上增加 Qt、rqt、web 或 RViz panel 实现完整 GUI
  • 实时控制：当前使用 fake controller，后续可接入真实 UR10 硬件控制器
